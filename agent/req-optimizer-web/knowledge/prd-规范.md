# PRD 写作规范（公司内部 v1.2）

## 必备六段式
1. 需求背景：必须包含业务价值与衡量指标（OKR/KPI）
2. 目标用户与场景：用户画像 + 至少 2 个典型使用场景
3. 功能需求 FR：每条 FR 必须含 描述 / 输入 / 输出 / 约束 / 异常分支
4. 非功能需求 NFR：性能、安全、可用性、兼容性、可观测性
5. 验收标准：使用 Given/When/Then 写法，每个 FR 至少对应 1 条 AC
6. 风险与开放问题：列出当前模糊点，并附"建议澄清问题"

## 编号规则
- FR 使用 `FR-编号 + 简短标题`，编号从 1 开始递增
- AC 使用 `AC-编号`，与 FR 不必一一对应，但需在描述里注明覆盖的 FR
- NFR 使用 `NFR-类别-编号`，例如 `NFR-性能-1`

## 最常见的 6 个反模式
1. "页面要好看" —— 没有具体设计稿、视觉规范、参考竞品
2. "越快越好""下周能上吗" —— 没有具体上线时间与里程碑
3. "要安全一点" —— 没有指定加密算法、合规要求
4. "兼容主流浏览器" —— 没有明确版本号
5. 没写"异常分支"的 FR
6. 验收标准用"页面正常显示"这种不可测语句
