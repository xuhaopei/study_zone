## 1. 需求背景

### 1.1 业务背景
当前 `study_zone` 缺少统一的账号登录入口，用户访问受限内容（收藏、学习记录、付费课程）时无身份凭证，转化漏斗从"浏览"到"留存"在入口处断裂。本次需求先解决"登录"这一最基础、最关键的入口能力，为后续的"学习进度同步""个性化推荐"等模块铺路。

### 1.2 业务价值与衡量指标（KPI / OKR）
| 指标 | 基线（上线前） | 目标（上线后 30 天） | 说明 |
| --- | --- | --- | --- |
| 登录转化率（到达登录页 → 登录成功） | 0%（无入口） | ≥ 60% | 衡量入口易用性 |
| 登录 P95 耗时 | N/A | ≤ 500ms | 接口层 NFR |
| 登录页 LCP | N/A | ≤ 2.5s | Web Vitals |
| 暴力破解相关安全事件 | N/A | 0 起 | 安全基线 |
| 登录相关 P0/P1 故障 | N/A | 0 起 | 可用性 |

> 业务 OKR 表述：O1 —— 完成账号体系从 0 到 1 建设；KR1 登录转化率 ≥ 60%；KR2 登录 P95 ≤ 500ms 且安全事件 0 起。

## 2. 目标用户与场景

### 2.1 用户画像
- **P1 学习者小白（主力）**：18–30 岁，主要通过手机端访问；账号通常只用手机号注册，期望"输手机号 + 验证码"一步到位。
- **P2 资深学习者**：已有第三方账号（微信/邮箱），对登录方式敏感，希望同时支持账号密码与手机号。
- **P3 内部运营 / 测试**：使用账号密码高频登录，需要"记住我"和快捷键盘回车提交。

### 2.2 典型场景
- **场景 S1：手机号一键登录**（P1，主链路）
  1. 用户在登录页输入手机号 → 点击"获取验证码" → 输入 6 位短信验证码 → 登录成功 → 跳转首页。
  2. 30 秒内未收到验证码可"重新发送"，60 秒内仅能发 1 次。
- **场景 S2：账号密码登录**（P2 / P3）
  1. 用户输入账号（手机号 / 邮箱）+ 密码 → 勾选"记住我" → 提交 → 跳转首页。
  2. 连续输错密码 5 次后账号被锁定 15 分钟，给出统一提示"账号或密码错误，请稍后再试"。

## 3. 功能需求（FR）

> 每条 FR 含：描述 / 输入 / 输出 / 约束 / 异常分支。

### FR-1 账号密码登录
- **描述**：用户通过账号（手机号或邮箱）+ 密码完成登录。
- **输入**：`account`（手机号 / 邮箱，非空）、`password`（非空，长度 8–32）、可选 `rememberMe`（bool）。
- **输出**：成功 → 颁发 Access Token + Refresh Token，跳转首页；失败 → 统一错误提示。
- **约束**：
  - 密码前端 SHA-256 一次后随 HTTPS 传输，后端用 **bcrypt（cost ≥ 10）** 校验；
  - 失败提示统一文案"账号或密码错误"，**禁止**区分"账号不存在"与"密码错误"；
  - 服务端不存明文密码，禁止 MD5/SHA1 摘要。
- **异常分支**：
  - 连续失败 5 次 → 账号锁定 15 分钟，期间返回 `ACCOUNT_LOCKED`；
  - 同一 IP 1 分钟内 ≥ 20 次失败 → 强制触发图形/滑块验证码（见 FR-4）；
  - 账号不存在 / 密码错误 / 账号已停用 → 统一文案 `LOGIN_INVALID`。

### FR-2 手机号短信验证码登录
- **描述**：通过手机号 + 短信验证码登录，未注册用户登录即视为注册（账号合并策略见 FR-6）。
- **输入**：`phone`（11 位国内手机号）、`code`（6 位数字）。
- **输出**：成功 → 颁发 Token，跳转首页。
- **约束**：
  - 验证码有效期 ≤ 24h，单次有效，使用后立即失效；
  - 同一手机号 60 秒内仅可获取 1 次，1 天内最多 10 次；
  - 发送行为与登录结果均需留痕。
- **异常分支**：
  - 验证码错误 / 过期 → `CODE_INVALID`；
  - 手机号格式非法 → `PHONE_INVALID`；
  - 当日获取次数超限 → `CODE_RATE_LIMIT`。

### FR-3 防暴力破解
- **描述**：账号、IP、IP+UA 三个维度组合限流。
- **输入**：登录请求上下文（账号、IP、UA）。
- **输出**：触发策略时返回对应错误码或强制验证码。
- **约束**：
  - 同账号 5 次失败 → 锁定 15 分钟；
  - 同 IP 1 分钟内 ≥ 20 次失败 → 触发图形/滑块验证码（FR-4）；
  - 验证码通过后才允许再次提交凭据。
- **异常分支**：锁定期间登录 → `ACCOUNT_LOCKED`（不暴露剩余时长精确值）。

### FR-4 图形 / 滑块验证码
- **描述**：高风险场景下强制二次校验。
- **输入**：用户在登录页拖动滑块 / 识别图形。
- **输出**：下发短期 `captcha_token`（≤ 5 分钟，仅用于本次登录请求）。
- **约束**：服务端校验，失败 3 次强制刷新；与登录接口在同一会话中绑定。
- **异常分支**：token 过期 / 失败 → `CAPTCHA_INVALID`，需重新验证。

### FR-5 会话管理与"记住我"
- **描述**：基于 Access Token + Refresh Token 的双 Token 方案。
- **输入**：登录成功的凭据。
- **输出**：
  - Access Token 有效期 ≤ 2h（前端内存存储）；
  - Refresh Token 有效期 ≤ 30d，仅在"记住我"勾选时持久化，Cookie 属性 `HttpOnly + Secure + SameSite=Lax`。
- **约束**：
  - Token 绑定设备指纹 / UA，登出时**双端失效**（前端清缓存 + 服务端吊销 Refresh Token）；
  - 同账号最多保留 5 个活跃会话，超过踢掉最早一个。
- **异常分支**：Access 过期 → 静默刷新；Refresh 过期 → 强制重新登录。

### FR-6 未注册手机号账号合并与同意条款
- **描述**：手机号未注册时登录成功后自动创建账号。
- **输入**：`phone`、`code`、是否勾选《用户协议》《隐私政策》。
- **输出**：创建用户并登录。
- **约束**：
  - **默认不勾选**同意，需用户主动勾选；
  - 隐私政策需列出"手机号用于登录与风控""短信由 XX 供应商发送"等字段用途；
  - 手机号属于敏感个人信息，**单独同意**且加密存储。
- **异常分支**：未勾选 → `CONSENT_REQUIRED`，阻断登录。

### FR-7 多端登录设备管理（基础版）
- **描述**：用户可在"账号与安全"页查看活跃设备并远程登出。
- **输入**：登录态。
- **输出**：设备列表 + 远程登出接口。
- **约束**：本期仅展示设备名/最近登录时间/操作系统，远程登出走 FR-5 的 Token 吊销。

### FR-8 登录页 UI 与交互
- **描述**：登录页默认提供"手机号登录"Tab 与"账号密码登录"Tab 切换。
- **输入**：用户输入。
- **输出**：登录页 DOM。
- **约束**：
  - 提供设计稿源文件（Figma 链接，里程碑 M2 提交），参考竞品：飞书登录、掘金登录；
  - 支持回车提交、错误就近提示、密码可显示切换；
  - 移动端适配 ≤ 375px 视口。

## 4. 非功能需求（NFR）

### NFR-性能-1
- 登录接口 P95 ≤ 500ms，P99 ≤ 1s（P95 测算覆盖 P1 主力用户的弱网环境）。
- 登录页 Web Vitals：LCP ≤ 2.5s，INP ≤ 200ms，CLS ≤ 0.1。
- 客户端到首字节 TTFB 在 4G 条件下 ≤ 600ms。

### NFR-性能-2
- 单实例 QPS 评估公式：`日活 × 人均登录次数 / 86400 × 安全系数 5`，容量评估与压测报告需在 M3 完成。

### NFR-安全-1（密码与传输）
- 客户端→服务端全程 HTTPS（TLS 1.2+）；
- 服务端**严禁**使用 MD5/SHA1/明文存储密码，统一使用 **bcrypt（cost ≥ 10）** 或 **argon2id**；
- 密码复杂度：长度 ≥ 8，至少含字母 + 数字两类，禁止 top1000 弱密码字典。

### NFR-安全-2（防暴力破解）
- 同账号 5 次连续失败 → 锁定 15 分钟；
- 同 IP 1 分钟内 ≥ 20 次失败 → 强制图形/滑块验证码；
- 失败提示统一为"账号或密码错误"，不可暴露账号是否存在。

### NFR-安全-3（会话管理）
- Access Token ≤ 2h，Refresh Token ≤ 30d；
- "记住我"持久化 Refresh Token 必须使用 `HttpOnly + Secure + SameSite=Lax` Cookie；
- Token 绑定设备指纹/UA，登出双端失效；
- 同账号最多 5 个活跃会话。

### NFR-可用性-1
- 登录服务可用性 ≥ 99.9%（月度）；
- 短信通道需支持至少 2 家供应商热切换，主通道故障 30 秒内自动切到备用通道。

### NFR-兼容性-1
- 浏览器：Chrome ≥ 100、Edge ≥ 100、Safari ≥ 15、Firefox ≥ 100；
- 移动端：iOS Safari ≥ 15、Android Chrome ≥ 100；
- 微信内置浏览器（基础适配）；
- 屏幕：375px ~ 1920px 视口均无横向滚动条。

### NFR-可观测性-1
- 日志（结构化 JSON）+ 指标（Prometheus）+ 链路追踪（OpenTelemetry）三件套；
- 登录接口 5xx 错误率 ≤ 0.1%；
- 关键告警（登录失败率突增 3 倍、短信下发失败率 > 5%）**5 分钟内**触达值班群。

### NFR-合规-1
- 登录页首次使用前展示《用户协议》《隐私政策》，**默认不勾选**；
- 收集手机号前明示"采集字段、用途（登录 / 风控）、存储期限（账号存续期间）"；
- 满足 PIPL/GDPR 通用的"查询/更正/删除/导出"4 项权利；
- 手机号属于敏感个人信息，**加密存储 + 单独同意**；
- 关键操作（登录、改密、改手机、删除账号）日志保留 ≥ 180 天，**严禁记录密码、完整身份证号、银行卡号**。

## 5. 验收标准（AC / Given-When-Then）

> 每个 AC 标注覆盖的 FR 编号。

**AC-1**（覆盖 FR-1）
- **Given** 已注册用户 `u1`，密码 `Passw0rd!`，且账号未被锁定
- **When** 在登录页输入正确账号密码并提交
- **Then** 跳转首页，接口返回 200 + Access/Refresh Token，耗时 P95 ≤ 500ms

**AC-2**（覆盖 FR-1 / FR-3）
- **Given** 用户 `u1` 连续输错密码 4 次
- **When** 第 5 次再输错
- **Then** 返回 `ACCOUNT_LOCKED`，账号锁定 15 分钟；前端提示"账号或密码错误，请稍后再试"，**不出现**"账号不存在"字样

**AC-3**（覆盖 FR-1 / NFR-安全-2）
- **Given** 攻击者从同一 IP 在 1 分钟内提交 20 次失败登录
- **When** 第 21 次请求到来
- **Then** 服务端返回 `CAPTCHA_REQUIRED`，登录页强制出现图形/滑块验证码

**AC-4**（覆盖 FR-2）
- **Given** 用户输入合法手机号 13800001111 并点击"获取验证码"
- **When** 60 秒内再次点击"获取验证码"
- **Then** 第二次请求被拒绝，返回 `CODE_RATE_LIMIT`；提示"请稍后再试"

**AC-5**（覆盖 FR-2 / FR-6）
- **Given** 手机号 13800002222 未注册，且未勾选《用户协议》
- **When** 用户输入正确验证码并提交
- **Then** 阻断登录，返回 `CONSENT_REQUIRED`；勾选后再提交可成功登录并自动创建账号

**AC-6**（覆盖 FR-5 / NFR-安全-3）
- **Given** 用户 A 勾选"记住我"登录成功后获取到 Refresh Token
- **When** 切换设备用同一个 Refresh Token 发起刷新
- **Then** 刷新失败并返回 `TOKEN_DEVICE_MISMATCH`，强制重新登录

**AC-7**（覆盖 FR-5）
- **Given** 同一账号已在 5 个设备登录
- **When** 第 6 个设备登录成功
- **Then** 最先登录的设备被踢下线，对应 Refresh Token 立即失效

**AC-8**（覆盖 FR-6 / NFR-合规-1）
- **Given** 新用户首次进入登录页
- **When** 页面渲染完成
- **Then** 顶部展示《用户协议》《隐私政策》入口且"同意"复选框默认**未勾选**

**AC-9**（覆盖 FR-8 / NFR-性能-1）
- **Given** 用户在 4G 网络下打开登录页
- **When** 首屏渲染完成
- **Then** LCP ≤ 2.5s，CLS ≤ 0.1，INP ≤ 200ms

**AC-10**（覆盖 NFR-兼容性-1）
- **Given** 使用 Chrome 100 / Safari 15 / iOS 15 / 微信内置浏览器
- **When** 走完"手机号登录"完整流程
- **Then** 登录成功，无控制台报错，无横向滚动条

## 6. 风险与开放问题

| # | 风险 / 开放点 | 影响 | 建议澄清问题 |
| --- | --- | --- | --- |
| R1 | 短信通道选型与单条成本未确认 | 决定 NFR-可用性-1 的双供应商方案 | 短信供应商是否已对接？是走阿里云 / 腾讯云 / 自建？预算单价上限？ |
| R2 | 第三方登录（微信、Apple）是否在本次范围 | 影响 FR 拆分与里程碑 | 本期是否只做账号密码 + 手机号？第三方登录放下一期？ |
| R3 | 未注册手机号登录即注册 vs 强制走"注册"流程 | 影响 FR-6 文案与转化 | 业务希望"低门槛登录"还是"显式注册"？是否需要单独的注册页？ |
| R4 | "登录要快速""界面要友好美观"无量化基线 | 验收主观化 | 是否有对标产品（如：飞书、掘金）？视觉风格走品牌指南 v1 还是新设计？ |
| R5 | 国际版是否在范围 | 决定是否需要海外手机号正则 + 短信通道 | 是否支持 +86 以外手机号？是否走 GDPR 严格模式？ |
| R6 | 失败次数统计维度（账号 vs IP） | 影响误伤率与安全强度 | 是否对内网 / 出口 NAT IP 做白名单？是否区分"密码错"与"账号不存在"两种计数？ |
| R7 | "记住我"刷新 Token 是否要支持轮换（rotation） | 决定 NFR-安全-3 强度 | 每次刷新是否签发新的 Refresh Token 并吊销旧的？ |
| R8 | Token 存储方式：Cookie 方案 vs 前端内存 + Refresh Cookie | 影响 XSS 暴露面 | 前端框架是 SPA 还是 SSR？是否允许前端 JS 接触 Access Token？ |

### 6.1 建议里程碑
- **M1（+1 周）**：需求评审通过、R1/R3/R5 澄清完成；
- **M2（+2 周）**：FR-1 / FR-2 / FR-3 / FR-6 主链路开发 + 联调，登录页设计稿定稿；
- **M3（+1 周）**：压测报告、可观测性面板上线、安全自查；
- **M4（+0.5 周）**：灰度 5% → 全量。
